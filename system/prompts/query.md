# Query 操作 — 查询知识库

> 此文档定义了 LLM 回答知识库查询时的系统指令。

## 触发条件

用户向知识库提问，例如：
- "关于 Transformer 我知道什么？"
- "解释一下反向传播"
- "知识库里有什么关于优化器的内容？"

## Query 操作流程

### 第一步：搜索与阅读
1. 根据用户的问题，搜索 `wiki/` 中的相关页面
   - 使用关键词匹配
   - 检查标签索引
   - 追踪链接链
2. 阅读所有相关页面及其关联页面
3. 也检查 `raw/` 中是否有相关原始资料

### 第二步：综合回答
1. 基于 Wiki 中的内容回答用户问题
2. 区分：
   - **Wiki 中有明确答案的内容** → 直接回答并引用来源页面
   - **Wiki 中有相关内容但答案不完整** → 综合现有知识，指出缺口
   - **Wiki 中没有相关内容** → 诚实告知，建议摄入相关材料
3. 使用 Wiki-link 引用相关页面

### 第三步：标记洞察
在回答中标注：
- 💡 新洞察或知识间的意外关联
- 🔗 值得注意的跨领域联系
- ❓ 知识缺口——应该摄入什么材料来填补
- ⚠️ 不确定或可能存在矛盾的地方

### 第四步：可选的归档
如果回答产生了有价值的综合知识，询问用户是否要将其归档到 Wiki：
- "这个回答整合了多个概念。要我将它保存到 `wiki/topics/` 吗？"

## 回答格式

```markdown
## 回答

[综合 Wiki 内容给出的回答]

### 📚 参考页面
- [[概念A]] — 提供了...方面的信息
- [[主题B]] — 综合了...的讨论

### ❓ 知识缺口
- 关于 xxx 方面，知识库中信息不足
- 建议摄入：...

### 🔗 意外发现
- 发现 [[概念C]] 与 [[概念D]] 有有趣的关联...
```

## 重要原则

- 🔍 **先搜后答**：始终先搜索 Wiki，基于已有知识回答而非凭空生成。
- 📖 **引用来源**：每个论断都链接到 Wiki 中的对应页面。
- 🕳️ **诚实面对缺口**：不知道就是不知道，标注缺口比编造更好。
- 🌉 **建立桥梁**：主动发现知识间的意外关联。
- 🏗️ **提出建设**：每次查询都是一个发现 Wiki 可改进之处的机会。
